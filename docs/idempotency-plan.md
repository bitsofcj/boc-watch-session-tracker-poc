# Idempotency Plan

---

## The Problem

The PRD explicitly states that Operations does not want to lose events during traffic spikes. The complementary risk — which the PRD does not mention but is equally real — is **duplicate events from client retries**.

When the SDK sends an event and the network fails before a response arrives, the SDK has no way to know whether the server received it. The correct behaviour is to retry. Without server-side deduplication, that retry produces a duplicate event record, which corrupts viewer counts and session history.

At hundreds of thousands of concurrent viewers sending heartbeats every 10–30 seconds, retries are not an edge case. They are a routine part of operating over real networks.

---

## The Current State

The top-level `eventId` field in the PRD's event schema is a session-scoped sequence identifier — `evt-1`, `evt-2`, `evt-3` — incrementing with each transmission from a given client session:

```json
{
  "eventId": "evt-3",
  "sessionId": "abc-123",
  "eventType": "heartbeat",
  "payload": {
    "eventId": "event-2026-wrestling-finals",
    ...
  }
}
```

Both clients implement this today. The server stores this field but does not enforce uniqueness on it. Duplicate transmissions produce duplicate rows in the events table.

---

## Deriving the Idempotency Key from Existing Fields

Because `eventId` is a session-scoped sequence (`evt-1`, `evt-2`, ...), it will repeat across different sessions — two sessions will both have an `evt-1`. But combined with `sessionId`, the pair is globally unique. **No new field is needed.** The combination of `sessionId + eventId` is already a globally unique key:

- `eventId` alone repeats across sessions
- `sessionId` alone is not unique per event
- `sessionId + eventId` together are globally unique, as long as the SDK never reuses an `eventId` for a different event within the same session

This is the preferred path. It requires no SDK contract change and no new fields — only a composite unique constraint on the events table and a change to the insert query.

### What changes on the server

**1. Schema — add a composite `UNIQUE` constraint on `(sessionId, eventId)`**

```sql
CREATE TABLE IF NOT EXISTS events (
  eventId        VARCHAR NOT NULL,
  sessionId      VARCHAR NOT NULL,
  eventType      VARCHAR NOT NULL,
  eventTimestamp VARCHAR NOT NULL,
  receivedAt     VARCHAR NOT NULL,
  payload        VARCHAR NOT NULL,
  UNIQUE (sessionId, eventId)
)
```

**2. Insert — use `ON CONFLICT DO NOTHING`**

```sql
INSERT INTO events (eventId, sessionId, eventType, eventTimestamp, receivedAt, payload)
VALUES (?, ?, ?, ?, ?, ?)
ON CONFLICT (sessionId, eventId) DO NOTHING
```

If the same `sessionId + eventId` pair arrives twice, the second insert is silently discarded. The server still returns `202 Accepted` — from the SDK's perspective the event was accepted, which is correct.

**3. Session upsert — already idempotent**

The `sessions` table already uses `INSERT ... ON CONFLICT (sessionId) DO UPDATE`. Replaying the same event against the same session produces the same session state. No change needed there.

---

## If the SDK Cannot Guarantee Unique `eventId` Values

If it turns out the SDK does not send a unique `eventId` per transmission — or sends no `eventId` at all — the composite key approach breaks down. In that case, the cleanest solution is a dedicated field with an explicit contract:

| Field | Type | Required | Description |
|---|---|---|---|
| `idempotencyKey` | string | yes | UUID generated by the SDK immediately before each HTTP call. Resent unchanged on retry. Never reused for a different transmission. |

The server would enforce uniqueness on `idempotencyKey` alone, and the composite `sessionId + eventId` approach would be abandoned. This requires a coordinated SDK contract change.

The v1 clients in this repo use a session-scoped incrementing counter (`evt-1`, `evt-2`, ...) for `eventId` today, which satisfies the composite key approach directly.

---

## Retention and Expiry

Idempotency keys in the events table are retained for the life of the event record. Since ClickHouse (the production store) retains raw events for analytical purposes, the keys are available for the full retention window at no additional cost. No separate key store or TTL mechanism is needed.

---

## What This Does Not Cover

**SDK-side retry logic.** The server enforcing uniqueness does not automatically give the SDK a retry loop. The SDK needs to implement exponential backoff with a fixed number of retries, resending the same `eventId` on each attempt. This is a coordinated SDK + server change.

**Out-of-order delivery.** Idempotency keys prevent duplicates but do not enforce ordering. Two heartbeats arriving out of order will both be stored; `eventTimestamp` is used for ordering in queries, not insertion order.
